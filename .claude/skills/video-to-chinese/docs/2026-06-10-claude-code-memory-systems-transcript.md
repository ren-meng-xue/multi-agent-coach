# 原始字幕（中译）：Every Claude Code Memory System Compared (So You Don't Have To)

来源：https://www.youtube.com/watch?v=UHVFcUzAGlM&t=305s
作者：Simon Scrapes
字幕轨道：en（翻译为简体中文）

---

**[00:00:01]**
现在，如果你搜索"最佳 Claude Code 记忆系统"，你会被彻底淹没。Mem Zero、Carpathia 的 Obsidian Wiki、Open Claude Hermes、Mem Palace、Light RAG、Claude Mem……越来越多，越来越难跟上。但在花了好几个小时研究之后，我发现这些根本不是竞争关系的工具——它们只是针对不同使用场景的不同记忆方式。我之所以深入研究这个问题，是因为我一直在构建自己的 Agentic 操作系统，本质上是一个贯穿我所有工作的"业务大脑"。所以我需要搞清楚，哪些真正有效、哪些能扩展，哪些只是给现有系统增加复杂度。

在这个视频里，我会带你了解六个层级的记忆系统，它们层层叠加，最终达到能跨所有 AI 工具使用的系统。看完之后，你会清楚地知道哪个适合你的场景，哪些可以完全忽略。

**[00:01:00]**
但在正式开始之前，有一件事我希望你先了解，因为它会让接下来的内容更容易理解——每一个记忆系统都在回答同一个问题：当你给 Claude Code 一个任务时，它如何在正确的时间拉取正确的上下文？

所以我要展示的每个层级之间的差异只有两点：第一，记忆存在哪里——存储机制和文件结构，比如是 Markdown 还是向量；第二，Claude 如何获取那些数据，也就是检索阶段。有些内容会自动复制进 Claude 的上下文，有些会存在数据库里按需搜索，有些在本地，有些在云上，等等。现在开始讲第一层级。

**[00:01:17]**
第一层级是 Claude Code 内置的原生功能。大多数人都听过 claude.md 文件，有些人听过 memory.md，但很少有人真正同时正确使用两者。先说 claude.md——这个你们大多数人已经知道，我快速过一下。它本质上是项目文件夹里的一个纯 Markdown 文件，你在里面存放规则、品牌信息、与客户的工作方式、代码风格等，并且每次在终端启动会话时都会自动加载。你可以把它想象成放在上下文开头的笔记，类似于给 Claude 的系统提示。

它可以放在 Claude 项目文件夹级别（传递给所有 Claude 实例）、根文件夹级别（如果多个项目在同一文件夹下），或者在单个项目级别放置特定的项目说明。层级越低，仍然会继承父级 claude.md，但有冲突时，本地 claude.md 的规则优先。

**[00:02:32]**
但是，大多数人在 claude.md 上犯的最大错误——我见过很多次——就是塞进去太多内容。把完整的品牌指南、语调文档、完整的客户列表全放进去。想想看，每次 Claude 启动会话，它都要把所有这些读一遍，消耗大量上下文窗口。这些内容固然重要，但只有在需要时才应该加载进来。

这正是我们要解决的问题，叫做"上下文腐化"（Context Rot）——随着加载的上下文越来越多，AI 和 LLM 无法 100% 召回其中所有信息。通用原则是：**保持 claude.md 在 200 行以内**。如果有更多内容，比如品牌声音文档，就放在单独的外部文件里，在 claude.md 中引用，只在需要时才拉取。

**[00:03:00]**
Claude Code 内置的第二个记忆系统叫做自动记忆（Auto Memory），会生成 memory.md 文件。你可以在现有项目文件夹里输入 `/memory` 来查看它。你会看到几种不同的东西：如果你用了单独的记忆系统，会有导入记忆；你还有项目和用户记忆，也就是不同层级的 Claude MD 文件。打开自动记忆文件夹，你会看到每个项目都有一个 memory.md 文件。

打开这些文件，里面有很多不同记忆的索引，Claude Code 可以引用并打开其他文件。它并不是把大量记忆全堆在一起，而是说：在项目级别，有项目级别的具体反馈；在更高层面，有各种类别的反馈，保存在单独的文档里。这完全符合我们说的原则：不要在单个文件里塞超过 200 行，永远引用单独的文件，把 memory.md 或 claude.md 当索引，指向其他文件。

**[00:05:04]**
举个例子，有一个关于"项目结构反馈"的文件。二三级项目归属于 project/briefs 下。然后有一个 feedback-project-structure.md，只有在 Claude 理解到我们在看项目结构时才会加载。这就是你随时间给它的反馈目录，或者它从你的工作方式中理解到的内容。

为了进一步说明，去一个空文件夹，你会看到里面没有 memory.md 或 claude.md。如果运行 /memory 命令，打开自动记忆文件夹，因为是按项目级别管理的，所以没有记忆整合。你不需要自己创建 memory.md 文件——它在后台静默记录你的任务和反馈，并为我创建了所有这些索引文件。

**[00:05:45]**
Claude.md 和 memory.md 就是 Claude 今天拥有的。Anthropic 显然也在自己解决这个问题——几周前 Claude Code 的源代码意外泄露，人们在代码里发现了一个叫 Kairos 的系统或框架。看了代码后，这是一个未发布的、始终运行的守护进程，它持续监视你的项目，决定什么值得记住，然后在你睡觉时整合旧笔记。目前还没有出现在任何公开版本里，但可以确定 Anthropic 在后台正在努力解决上下文腐化问题并改进原生记忆系统。只会越来越好。

第一层级的设置已经为你完成了，在后台运行，完全免费，你只需要正确使用它们。

**[00:06:20]**
如果你不断重复告诉 Claude 它本应该已经知道的信息，那就是第二层级登场的时候了。第二层级提高了在正确时间加载记忆文件的可靠性，同时也改善了 Claude 内置的结构。

我找到了一篇 John Connolly 写的文章，他基于 Pavel Huryn 的概念——科技圈的大名人，2 月 18 日写的"如何给 Claude Code 记忆"，把这个粘贴到你的项目 claude.md 里。注意，这是在 Claude 宣布自动记忆功能之前写的，但它现在做的是在 Claude 的结构基础上进一步扩展。结合 John 的方案，我们就能在每次会话开始时自动注入记忆。

本质上，我们在 claude.md 里添加了一大段关于如何管理记忆和规则的提示。

**[00:07:00]**
结构化记忆系统根目录在 .claude/memory。我们已经看过 Claude 自动记忆系统，但这里有一个明确的结构：memory.md、general.md（存储跨项目的事实、偏好、环境设置），还有领域专属知识，即 domain/topic，每个话题可以有自己的 Markdown 文件，一个文件对应一个话题；工具也类似，可以有 slack.md 这样的文件，存放特定工具的配置、变通方案和边界情况。

我们把这些加到 claude.md，然后进入计划模式，让它为我们的项目建立一个记忆结构，之后可以自动注入到每次会话。

**[00:07:37]**
这通过 Claude Code 的会话启动钩子（session start hook）实现，确保每次打开新终端或启动子 agent，都自动加载 memory.md 索引到会话文件。如果你没用过钩子：它就是一个小脚本，在特定时刻（比如会话启动）把上下文注入 Claude。我们不是加载全量上下文，只加载 memory.md 这个索引文件。

第二个令人兴奋的地方是团队共享的潜力——想象一下，把我们讨论的领域文件在团队成员间同步。每个人工作在不同项目上，可以把与该上下文相关的记忆文件保存到我们都能访问的共享文件夹里。

**[00:08:53]**
这是 Claude Code 在某个领域或工具上真正发挥作用所需的具体细节。现在来演示一下如何设置。我们从描述里的链接复制第一部分的整个 Markdown 段落，粘贴到 claude.md 里。它会创建全局记忆目录结构，更新 claude.md 文件，初始化项目 memory.md 文件。

当我说"reorganize memory（整理记忆）"时，它会读取所有记忆文件，删除重复和过时条目，合并属于同一主题的条目，拆分覆盖太多话题的文件，按日期重新排序，并更新 memory.md 索引。

**[00:10:08]**
我们不是直接把这个放进 claude.md，而是打开一个新终端窗口，切换到计划模式，粘贴整个提示。Claude 会根据已有内容规划记忆文件结构，以及需要在 claude.md 里添加什么。Claude 完成第一部分的计划后，我们再次进入计划模式，运行"reorganize memory"来整理记忆文件，看看它从我们之前的项目里构建出了什么。

很简单：等待执行、检查计划、构建、然后执行"reorganize memory"推送所有改动。

**[00:10:45]**
你可以看到它现在在探索记忆系统状态，然后返回计划——"结构化持久记忆管理系统，为 Claude Code 建立全局结构化记忆系统，让知识跨会话和项目持久保存"。当前根目录的 claude.md 存在但为空，Claude memory 不存在，28 个项目中只有一个有 memory/memory.md。它沿着父目录树一直检查，了解最高层级存在什么、会向下继承什么。

**[00:11:23]**
它会创建包含 general、tools、domain 的全局记忆目录结构，用五个部分更新 claude.md（记忆管理、全局记忆、repo 记忆自动初始化等），遍历所有文件，创建记忆系统结构，然后我们用它来整理记忆。

之后我们会设置钩子，让每次加载会话时自动注入记忆索引。

**[00:12:39]**
在所有 28 个项目中，它为每个项目创建或整合了一个 memory.md 文件。现在运行"reorganize memory"（保持在计划模式），它会规划整理记忆的方案，开始用相关内容填充 memory.md 和 claude.md 文件。

**[00:13:00]**
同时，来看第二步——引入钩子。这个钩子在每次会话的第一个工具调用之前注入你的项目记忆和全局 memory.md 索引，让每次启动 agent 或子 agent 时都知道自己有哪些记忆可以访问。

它会在 settings.json 里创建钩子，设置成 pre-tool-use 时运行 pre-tool-memory.sh，该脚本加载我们刚创建的记忆文件。

**[00:13:55]**
然后它更新 claude.md——不再依赖 Claude 在会话开始时主动读取 claude/memory/memory.md，而是通过钩子自动注入。这就不依赖 Claude 自己读取并采取行动了。

John 5 天使用后，记忆文件数量大幅增加（增长 90%，行数明显增多），但 memory.md 或 general.md 本身没有增长多少——因为它们只是指向正确工具、领域和循环文件的索引。

**[00:14:30]**
我让它整理记忆，结果非常好：它扫描了我的 AgentiCoS 文件夹，发现了 11-12 个空记忆文件（空模板、空会话），说这些应该删掉，加载它们没有意义。第二阶段修剪空会话，第三阶段解决过期的未完成线程——读取每个日常记忆里的具体线程，找到后续发生的事情来更新它们。

**[00:15:11]**
然后它为我们做过的所有项目添加交叉引用，更新 general.md，更新 memory.md 里的索引，最后验证。结果：删除了 13 个文件，修剪了 6 个会话，完成了 9 个未完成线程，添加了 10 个交叉引用，general.md 增加了 3 个模式，保留了 12 个有实质内容的日常文件，清除了客户文件夹里所有空记忆目录。

**[00:16:27]**
对很多人来说，这就够了。但用超过几周之后，你会遇到问题：文件开始变乱，这个系统的关键词搜索在文件量小时有效，但随着增长和扩展会失效。这就是第三层级登场的时候。

触发时机：如果这两点同时成立——一是用 Claude Code 超过一个月且记忆文件已经超过几个；二是你曾问过 Claude 一个问题，你知道答案在笔记里但 Claude 找不到——那就该升到第三层级了。

**[00:17:43]**
第三层级的模板来自你可能熟悉的 AI agent——Open Claude。Open Claude 的文件结构来自那个独立的 Open Claude agent，不是 Claude Code，但它的记忆设计系统非常强大且简洁。

它有三层：

1. **memory.md**：长期持久事实——不常变的偏好、决策、业务事实，每次会话开始时加载。
2. **日常笔记文件**：每天一个文件，记录今天发生了什么、什么决定了、什么发布了、什么没奏效。今天和昨天的自动加载，更早的保留但不进上下文。
3. **Dreaming（可选）**：后台进程读取日常笔记，给它们打分，把反复出现的内容提升到 memory.md 长期记忆，陈旧内容被遗忘，避免撑爆上下文。

**[00:18:58]**
这给了我们一个机制：短期记忆通过 dreaming 决定什么嵌入长期记忆并馈入 Claude。由此引出一个叫 **MemSearch** 的框架，这是一个插件，把 OpenClaude 的框架移植到 Claude Code。由 Zilliz 构建——他们是最受欢迎的开源向量数据库之一的团队，深谙语义搜索之道。

他们把 OpenClaude 的记忆架构提取成一个独立库，两行命令即可接入 Claude Code。它保留了 OpenClaude 的所有优点：Markdown 优先哲学（所有内容可见、可读、可迁移到其他系统）、相同的 chunking 策略、相同的文件结构（memory.md + 日常笔记）。

**[00:20:13]**
工作方式：把你写的所有文档分块为语义向量，搜索时理解问题和存储内容的含义，而不只是关键词匹配。更进一步的是它用了钩子——自动把每个提示的顶部匹配注入进去。你一写提示，`user prompt submit` 钩子触发，把记忆文件里最相关的前三条直接注入 Claude Code 上下文。你不需要记得让 Claude 去搜，它会在你输入查询的时候自动注入最相关的记忆。

**[00:21:00]**
安装方式：`/plugin marketplace add zilliztech memsearch`，选择添加到本地文件夹；然后 `/plugin install memsearch`，只为这个仓库安装。它会生成 settings.local.json 文件，在其中启用 memsearch 插件。运行 `/reloadplugins` 激活，然后 Ctrl+C 两次重新打开 Claude Code。

验证方式：聊几次，然后列出 `.memsearch/memoryfiles` 下的文件，应该能看到今天的记忆文件。也可以用内置的 `/memory recall` 指令。

**[00:22:00]**
这个系统和我们在 Agentic OS 里用的非常相似——保存日常日志文件，加上可以在正确时间语义注入的长期记忆文件。

另一个值得了解但我认为不太适合大多数用途的替代品是 **Claude Mem**——自动捕获 Claude 在编码会话中的所有操作，用 AI 压缩，然后注入相关上下文到未来会话。它用的是 MCP 工具和三层存储模型（摘要、时间线、观察），功能比 MemSearch 多（仪表盘、团队协作、成本追踪、隐私标签），但对我们来说有点过度设计。

**[00:22:43]**
Claude Mem 基于 MCP，意味着 Claude 需要主动决定调用搜索工具，而不像 MemSearch 那样从本地数据库自动注入。另一个关键区别：Claude Mem 在后台存储一切（不保留可读文本），而 MemSearch 以纯 Markdown 保存，你可以随时回去读。

**[00:23:10]**
第四层级是逐字逐句的对话回溯，使用 **Mem Palace** 框架。触发条件：你开始想"我知道我们几周前讨论过这个，但我记不清到底决定了什么"。第三层级能找到笔记和摘要，第四层级能找到做决策时用的**确切措辞**，而且一切存储在本地。这是一个完整的 RAG 系统，免费，目前号称是有史以来发布的基准分数最高的记忆系统。

**[00:23:59]**
因为逐字存储，没有任何东西会被总结，理论上什么都不会丢失。索引用 AAAK 语言（一种 LLM 可以快速扫描的密集符号方言）编写，它是一系列指向特定数据位置的指针。

举例："我儿子叫 Noah，9 月 12 日满 6 岁，他喜欢恐龙。"这句话会被索引为 Noah、儿子、6 岁、生日，然后存到"42 号翼、11 号房、007 号抽屉"。记忆宫殿的结构：翼 = 人物、项目、话题；房间 = 某天、某会话或线程；壁橱 = 主题或束；抽屉 = 逐字文本。

**[00:25:15]**
因为存在这个符号索引里，模型可以在一次扫描中浏览数千个抽屉，速度非常快。演示：有了 Mem Palace，说完某句话，它立刻存到某个抽屉；两周后提问，42 毫秒就能从那个抽屉里检索到原文。

**[00:26:30]**
技术架构：两个数据库——SQL 数据库追踪实体和关系，ChromaDB 向量数据库存储所有对话的可搜索分块。也通过钩子静默存储和索引，在会话结束、pre-compaction 时触发。一条命令安装，初始化后在 .mem-palace 文件夹创建翼/房/抽屉结构，在 settings.json 注册钩子，还可以用 mine 函数追溯处理历史会话。

**[00:27:46]**
有了 Mem Palace，Claude 可以用自然语言逐字搜索你有史以来的每一次对话，因为都被索引在后台。缺点是数据不是可读的 Markdown，但检索很快。而且到目前为止，所有东西都锁在本地机器上——ChatGPT 里的研究、手机上的头脑风暴都不在里面。这就是第五、六层级的用武之地。

**[00:29:01]**
**第五层级**和之前完全不同——不再是记忆对话，而是建立跨工具的知识库。适合场景：你定期消费想要保存和连接的信息——文章、视频、播客、客户笔记，你在大量阅读中不断迷失所学。你可以把这些连接成一个互联的知识库，更像是在你关心的话题上积累"第二大脑"，而不是针对一次性项目。如果你只是随意浏览内容、不需要回头查，可以跳过这一层。

**[00:30:18]**
最流行的是 **Andrej Karpathy 的 LLM Wiki**——一种用 Claude 为你关心的任何话题构建活体 Wiki 的模式。做法：创建两个文件夹。`raw/` 文件夹放源文档（文章、报告、播客转录、YouTube 字幕、PDF），Claude 只读不写这里；`wiki/` 文件夹由 Claude 完全掌管，它写所有文件、维护结构、随时更新交叉引用，你不直接写 wiki。整个系统只需纯 Markdown 文件，配合 Obsidian 可以得到知识图谱。

**[00:31:33]**
但老实说，我不太确定它在像 Business OS 或 Agent OS 这样的系统里有明显的直接用途——它更适合：你有一个"稍后阅读"的文件夹，里面堆满了你从未回看的 Slack 讨论、YouTube 视频、客户笔记，想看到它们之间的关联；或者你想针对某个特定主题做深度研究并创建互联 Wikipedia。

**[00:32:00]**
类似的托管替代品叫 **Recall**——安装浏览器扩展，保存文章、播客、PDF，它自动摘要、打标签、构建知识图谱，还有 AI 聊天界面，支持手机，有 MCP 接入。但缺点：数据在他们服务器上（你不拥有数据）；它为内容消费而建，不是运营记忆（适合"哪个视频讲了记忆"，不适合"3 月份我们对客户 X 的落地页做了什么决策"）；还有定价。

非技术用户、主要整理文章视频播客 → Recall 不错。想要最大控制权、自建 Wiki → LLM Wiki 更好。

**[00:34:04]**
还有 **LightRAG**——企业级知识图谱，实体提取、双层检索，研究级系统。对 99% 的人来说完全过度设计，不推荐。

**[00:34:30]**
**第六层级**是跨 AI 工具共享记忆，方案叫 **OpenBrain**（Nate Jones 提出）。目标是在不同 AI 工具间无缝切换——手机上的 ChatGPT、桌面的 Claude Code——所有 AI 实时看到同一记忆。最大优势：记忆存在你自己拥有的 Postgres 数据库里，当下一个重大 AI 工具出现时，直接接入已有数据库，无需迁移。

**[00:35:21]**
架构：一个数据库（Supabase 上的 Postgres）、一个聊天频道，任何 AI 都可以接入。数据库里有一张 thoughts 表，每行是一条记忆（文本分块 + 嵌入向量 + 标签 + 时间戳），用内置扩展处理语义搜索。Claude Code 通过 MCP 服务器连接 Supabase 的 Edge Function，ChatGPT、Claude Desktop、Cursor 都可以查同一个数据库。

缺点：设置时间长（30-45 分钟），比其他层级复杂。每次查询都要访问外部数据库，会增加延迟。不过每月费用极低（Supabase 免费层约 0.1-0.3 元/月）。

**[00:37:52]**
替代品 **Mem0**（mem0.ai）：另一个跨工具记忆层，资金雄厚、广泛使用（10 万+ 开发者），一分钟内上手。适合在 AI 产品里做生产级应用。缺点：数据在他们服务器上，你不拥有数据控制权。

**[00:39:07]**
**总结与选择建议：**

- **刚刚开始** → 不用管这些记忆系统，先把 claude.md 和 memory.md 用好（10 分钟，立竿见影）。
- **用了一段时间** → 升到第二层级，装 John 的钩子，大多数人到这里就够了。
- **积累了大量上下文、开始丢失历史决策** → 第三层级（MemSearch，语义搜索）或第四层级（Mem Palace，逐字检索）。
- **需要建立跨主题研究知识库** → 第五层级（LLM Wiki 或 Recall）。
- **需要跨 AI 工具共享记忆** → 第六层级（OpenBrain 或 Mem0）。

好消息：一到三层可以叠加运行，文件夹结构兼容，让 Claude Code 自己整合即可。

**[00:41:00]**
我个人在 Agentic OS 里实现到第三层级：遵循 OpenClaude 惯例，加上语义搜索能力和钩子，在正确时间注入正确上下文，改善了业务操作系统里的搜索和召回功能。
