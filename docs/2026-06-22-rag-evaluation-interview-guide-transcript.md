**[00:00:00]**

今天这道面试题是如何评估 RAG 系统的效果。这是一道高频题，而且是一道很容易答的太浅的题。很多人上来就说用 RAGAS 打个分就行了，但面试官真正想听的远不止这些。

没错，这题我面试的时候就被问过。当时我说了 faithfulness 和 answer relevance，面试官追问了一句：这些分数是谁打的？我一下就卡住了。这个追问非常关键，我们后面会专门讲。

**[00:00:29]**

先从最基本的开始：为什么评估 RAG 不能用传统的 NLP 指标，比如 BLEU、ROUGE 这些？这些不是做机器翻译和文本摘要用的吗，为什么不能用？三个原因：

第一，它们只比较字面重叠，不懂语义。RAG 的正确答案可以有很多种说法，两个同样正确的答案可能几乎没有词汇重叠，BLEU 算分就是零。

第二，它们完全无视上下文。BLEU 只看输出跟参考答案像不像，根本不管生成内容是不是从检索到的文档里来的。

第三，也是最致命的，它们无法检测幻觉。RAG 最核心的风险就是看起来有据可依，但实际在胡说，传统指标对此毫无感知。

所以传统指标只能测表面，测不了 RAG 真正需要的东西——忠实度和上下文利用率。

**[00:01:22]**

那有专门为 RAG 设计的指标吗？对，目前业界最广泛使用的是 RAGAS 框架定义的三个核心维度：Faithfulness（忠实度）、Answer Relevance（答案相关性）、Context Relevance（上下文相关性）。用大白话说，就是有没有胡说、有没有答对、有没有找对。

这三个怎么测？需要人工准备标准答案吗？这是 RAGAS 最大的卖点——三个都不需要人工标注，答案全部由 LLM 自动完成。

**[00:01:54]**

我一个一个讲，先说 **Faithfulness（忠实度）**。输入是问题、检索到的文档，还有模型生成的答案。

第一步，让 LLM 把答案拆成一条条独立声明。比如答案是"诺兰执导了奥本海默，基里安·墨菲主演"，那就拆成两条声明。

第二步，逐条拿去跟检索文档对比：这条声明有没有被文档支持？

第三步，算分：被支持的声明数除以总声明数。

等一下，如果检索到的文档本身就是错的呢？答案忠实地复述了错误信息，faithfulness 是不是还是满分？没错，这正是 faithfulness 的局限。它只管答案有没有从文档里来，不管文档本身对不对。文档写着"地球是平的"，答案忠实复述了，faithfulness 照样 1.0。

所以 faithfulness 不等于正确性，它只测忠实度，不测真实性。

**[00:02:49]**

那 **Answer Relevance** 呢？它的思路很巧妙，不需要检索文档，只需要原始问题和生成的答案。测法是让 LLM 看着答案反向猜：这个答案在回答什么问题？生成好几个问题，然后用 embedding 模型算这些反向问题跟原始问题的余弦相似度，取平均。

如果答非所问，比如你问发射日期，他答"这个火箭对天气研究很重要"，反向生成的问题就会跟原问题差很远，分就低。巧妙——不直接判断对错，而是用反向生成来间接度量。

**[00:03:28]**

那 **Context Relevance** 呢？这个评的是检索器的质量。输入是原始问题和检索到的文档，甚至不看模型答案。让 LLM 从文档里挑出对回答问题真正有用的句子，然后算比例：有用的句子数除以文档总句子数。检索到的内容越聚焦，分越高。

如果你检索了一大堆不相关的东西塞进上下文，即使最终答案碰巧对了，这个分也会很低，说明你的检索器在浪费 token。

**[00:04:03]**

好，三个指标我懂了，但这三个全是 LLM 自己在打分，LLM 评 LLM，这不是自己给自己判卷吗？哼哼，就是你面试时被追问的那个点。这确实是目前 RAG 评估最大的争议。

**[00:04:17]**

NVIDIA 去年做了一个大规模研究，让 54 个不同的 LLM 给将近 2000 个 RAG 输出打分，每个样本配三位人类标注者做对照，总共将近 6000 条标注。结果是 54 个 LLM 里有 36 个跟人类的相关性超过 0.8。不过论文进一步分析后发现，真正达到人类水平判断一致性的只有 27 个——高相关不等于高一致。

当时相关性最高的是 Llama 3 70B，达到了 0.88，但论文指出它属于"超一致性"类型，一致性甚至高于人类标注者之间，可能在过度简化细微判断。注意这是去年的数据，测的是当时的模型，不过结论的方向性——多数主流 LLM 跟人类判断大方向一致——应该还是成立的。

**[00:05:03]**

但是，但是，有三个已知偏差：

第一，**位置偏差**：把两个答案的顺序换一下，相当比例的判断会直接反转。

第二，**冗长偏差**：更长的答案更容易被偏好，不管内容质量。

第三，**自我偏好**：2025 年一项研究专门测了九个模型，包括当时的 GPT-4o 和 Claude 3.5 Sonnet，发现部分模型会系统性地给自己的输出打更高分。GPT-4o 和 Claude 3.5 Sonnet 就是典型。虽然现在模型已经迭代到更新版本了，但这个问题的结构性原因没有消失。

更有意思的是，他们还有"家族偏好"：Claude 系列的裁判倾向于给其他 Claude 模型也打高分，GPT 系列同理。所以选裁判时最好用跟被评估模型不同家族的。

**[00:05:51]**

那有什么办法缓解吗？还是说只能这么将就？不是将就，是在约束条件下的最优解。传统指标评不了语义，人工评不过来量，所以"LLM 当裁判 + 偏差缓解 + 人工校准"是目前的标准做法。具体有三类方案：

第一类，**工程手段**：AB 顺序各评一次取平均，解决位置偏差；评分标准里加简洁性维度，解决冗长偏差；用不同模型家族当裁判，解决自我偏好——比如生成用 GPT-4，评估用 Claude。

第二类，**微调专用小模型当裁判**：Stanford URM 就是这个思路，用 DPO 做裁判。

第三类，**人机混合**：大部分样本自动筛，高风险和边界 case 交人工审核。

**[00:06:43]**

好，指标讲清楚了，但面试官可能还会问：实际工程中你怎么做评估？不是说用哪个框架，而是具体的测试流程。这就是答题的第二个层次了。

**核心方法论是三层评估**：先分开评 retriever 和 generator，再端到端评。原因很简单——RAG 出了问题，可能是检索的错，也可能是生成的错。如果只看最终结果，你根本不知道问题出在哪。

**[00:07:08]**

黄金测试集——这不就又回到需要人工标注了吗？刚才不是说 RAGAS 不需要人工？好问题，RAGAS 的三个指标确实不需要人工标注，但分层测试需要，尤其是测 retriever 的时候，你得知道哪些文档是正确的，才能算 precision 和 recall。

所以黄金测试集就是一组你已经知道正确答案的数据，每条包含三个东西：用户问题、对应的正确文档是哪些，以及标准答案。一般需要 20~50 条，覆盖：常见问题、多文档综合题、知识库里没答案的题、还有信息过时或矛盾的题。

明白了——RAGAS 是自动评指标，黄金测试集是分层诊断用的，两个东西不矛盾。

**[00:07:55]**

有了测试集之后：

**第一层，单独测 retriever**：给他一个问题，看返回的 top-k 文档跟测试集里标注的正确文档对比。这一步完全不涉及 LLM，用的是传统信息检索指标：Precision@K（top-k 里有多少是相关的）、Recall@K（所有相关文档被找到了几个）、MRR（第一个相关文档排第几）、NDCG（考虑排序质量）。

**[00:08:21]**

**第二层，单独测 generator**：关键操作是跳过 retrieval，直接把测试集里标注的完美文档喂给 generator，看他的输出质量，用 faithfulness 和 answer relevance 这些指标。这样做的意义在于排除了检索错误的干扰。如果 generator 在拿到完美上下文的情况下仍然出错，说明问题在生成端——提示词不好，或者 LLM 能力不足，跟检索无关。聪明——把变量控制住了。

**[00:09:01]**

**第三层，端到端**：让 retriever 真实检索，加 generator 真实生成，用前两层的所有指标一起评，然后用一个诊断矩阵来定位问题：

- 端到端差，但 retriever 和 generator 单侧都好 → 问题在两者交互，比如上下文被截断了
- 端到端差 + retriever 差 + generator 好 → 检索问题，去调 embedding 或 chunking
- 端到端差 + retriever 好 + generator 差 → 生成问题，调 prompt 或换 LLM

**[00:09:35]**

那工具方面呢？面试官如果问你用什么工具做这些评估——这里要诚实，整个 RAG 评估领域都还不成熟。目前最常被提到的是 **RAGAS** 和 **DeepEval**：RAGAS 有 13000 多个 GitHub star，是行业默认选择，但注意默认不等于成熟，开发者社区对他吐槽不少，"让猴子管动物园"。DeepEval 有 14000 多 star，增长最快，月下载超过 250 万，杀手特性是原生 CI/CD 集成。说白了就是没有银弹。

**[00:10:11]**

那面试的时候被问到用什么工具，要不要直接这么说？建议这么打：真正靠谱的方案不是依赖单一框架，而是三层叠加：

第一层，用 RAGAS 或 DeepEval 做自动化初筛，快速发现明显问题；

第二层，积累领域定制测试集，把真实的失败 case 不断加进去，这才是团队最宝贵的资产；

第三层，人工抽检，高风险系统不可省略。

面试官听到这个会觉得你确实做过，不是纸上谈兵。

**[00:10:42]**

那给大家总结一下面试被问到这题，回答的框架是什么。面试回答可以按五个层次展开：

**第一层**，先说评估维度：忠实度、答案相关性、上下文相关性三大核心维度。

**第二层**，说方法论：分别评 retriever 和 generator，再端到端评，附带诊断矩阵。

**第三层**，承认 LLM-as-judge 的局限：自我偏好、位置偏差，以及缓解手段。

**第四层**，提工具，但要诚实：RAGAS 做快速实验，DeepEval 做 CI/CD 门禁，但都不是赢的。

**第五层（加分项）**，提到真正成熟的方案是自动化 + 定制测试集 + 人工校准，三层叠加。

达到第三层就及格了，达到第五层就是优秀。

**[00:11:31]**

最后再追一个问题，RAG 的七个失败模式，你没展开讲，这个面试会问吗？通常不会直接问，但如果你主动提到，会非常加分。快速说一下——学术上总结了七个失败点：

1. 知识库里根本没答案
2. 相关文档排名太低，没进 top-k
3. 上下文太长，被截断
4. 答案在文档里，但 LLM 没成功提取出来
5. 格式不对
6. 力度不当
7. 回答不完整

关键洞察是：没有任何单一指标能覆盖所有失败模式，这就是为什么需要多指标组合加分层测试的原因。
