# Unity Catalog：代理治理的基础设施

## 一句话总结

Unity Catalog 是 Databricks 的集中式数据目录，通过三层命名空间（目录→模式→对象）统一管理结构化数据、非结构化数据、向量数据库、函数工具和 AI 模型，是将四大治理支柱落地的技术基础。

## 核心观点

1. **Unity Catalog 是治理的物理基础** — 它不只是数据目录，而是将上一讲的四大治理支柱（生命周期管理、风险管理、安全、可观测性）在 Databricks 上的统一落地点，提供访问控制、审计追踪、数据血缘追踪和数据质量监控。

2. **三层命名空间：目录 → 模式 → 对象** — Unity Catalog 用元存储（Metastore）作为顶级容器，内部采用三层结构：Catalog（目录）→ Schema（模式/数据库）→ 具体对象（表、视图、卷、函数、模型）。这种层次结构是权限管理和治理的骨架。

3. **五类可治理对象覆盖 AI 全链路** — Unity Catalog 管理的对象不只是传统数据库资产：
   - **表（Table）**：结构化数据，行列格式
   - **视图（View）**：基于表的保存查询，可实现数据遮蔽/脱敏
   - **卷（Volume）**：非结构化数据（PDF、图片、视频等）
   - **函数（Function）**：可执行逻辑，直接作为 Agent 的工具/技能
   - **模型（Model）**：通过 MLflow 打包的 AI 模型，Agent 本身也注册为模型

4. **向量数据库继承 Unity Catalog 安全体系** — 基于 Databricks Vector Search 创建的向量索引（可来自表或卷）会自动继承 Unity Catalog 的所有权限和审计能力，AI 检索应用（RAG）无需单独设置安全层。

5. **最小权限访问是实验室的默认设计原则** — 课程演示中刻意使用自定义权限（而非"全部权限"），为开发者组仅授予：模式访问 + 创建模型/模型版本 + 读取表和视图 + 执行函数 + 创建表。这直接对应上一讲安全支柱中的"最小权限"原则。

6. **Agent 在 Unity Catalog 中以模型身份注册** — 代理（Agent）和微调模型统一通过 MLflow 打包后注册到 Unity Catalog，从而获得与普通数据资产相同的版本控制、访问控制和审计能力，这是"生命周期管理"支柱的关键实现机制。

7. **视图是 PII 数据治理的核心手段** — 对敏感数据（如社会安全号码、姓名）创建分析视图进行脱敏，Agent 只能通过视图查询数据，而非直接访问原始表，实现数据治理与功能需求的平衡。

## 可执行建议

- **用三层命名空间规划 AI 项目资产**：建立 `<项目名>.catalog` → `<业务域>.schema` → `tables/volumes/functions/models` 的层次，从一开始就纳入 UC 管理，而非事后迁移。
- **为 Agent 函数工具也注册到 UC**：不要把 Agent 工具写成游离的 Python 函数——注册为 UC Function，就能获得访问控制和审计能力。
- **永远通过视图暴露敏感数据**：原始表包含 PII 时，创建脱敏视图作为 Agent 唯一的查询入口；视图权限独立于原表，可做精细控制。
- **把 Agent 注册为 MLflow 模型**：这样 Agent 就能纳入 UC 的版本控制、访问控制和血缘追踪体系，和数据资产一起管理。
- **用自定义权限代替"全部授予"**：开发环境允许宽松权限快速迭代，staging/生产必须切到最小权限集，参考本课演示的五项权限清单。
- **向量索引优先用 Databricks Vector Search 创建**：这样自动继承 UC 安全体系，不用为 RAG 的检索层单独设置权限管理。

## 关键术语

| 术语                         | 说明                                                    |
| ---------------------------- | ------------------------------------------------------- |
| Unity Catalog（统一目录）    | Databricks 的集中式数据与 AI 治理平台                   |
| 元存储（Metastore）          | Unity Catalog 中数据的顶级容器，跨工作区共享            |
| 目录（Catalog）              | 三层命名空间的第一层，对应一个业务域或项目              |
| 模式（Schema）               | 三层命名空间的第二层，等同于传统数据库的 database       |
| 卷（Volume）                 | Unity Catalog 管理的非结构化数据对象（PDF/图片/视频等） |
| 函数（Function）             | UC 中的可执行逻辑单元，可直接绑定为 Agent 工具/技能     |
| 数据血缘追踪（Data Lineage） | 追踪数据从创建到消费的完整流转路径                      |
| 分析视图（Analytics View）   | 基于原始表创建的脱敏视图，是 PII 保护的核心机制         |
| 数据遮蔽（Data Masking）     | 对敏感字段（姓名/SSN 等）进行脱敏处理的技术             |
| MLflow 模型注册              | 将 Agent 打包并注册到 Unity Catalog 的标准方式          |
| 最小权限预设                 | UC 中可手动配置的细粒度权限集合，对应安全支柱的核心实践 |
