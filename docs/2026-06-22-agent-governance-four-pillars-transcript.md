那么当今构建代理的现实情况是怎样的？通常他们会先进行原型开发，因此大多数系统最初作为 POC 或演示版本启动。此时可行性优先于可扩展性和治理。以演示形式启动并非一定是坏事，直到性能开始优先于治理。

**[00:00:27]**

因此模型的准确性及运行指标优先于访问权限、控制与审计权限。我们不希望这些被性能所掩盖，尤其是在生产环境中出现问题时。我们知道唯一能有效排查问题的方法就是治理。拥有过高权限和访问权限是非常危险的。

**[00:00:48]**

当你准备将系统投入生产时，通常当开发者开始构建，他们会获得所有环境的广泛权限，而非基于角色的访问控制。接下来是当今构建代理的现实情况：审计追踪非常匮乏，团队无法准确追踪代理的操作及其访问内容。代理失败的位置导致调试变得极为困难，因此部署周期会延长三倍。

**[00:01:14]**

你可能有一个耗时 2~3 周构建的优秀 POC，然后将其交给团队投入生产，却需要八个月时间，因为他们必须重新调整所有安全和治理环节。你不愿承担这种额外的安全风险，因此最好从一开始就实践治理。

**[00:01:33]**

什么是数据治理呢？数据治理是政策、流程和标准的框架，确保数据准确、安全、可靠，并在其生命周期中得到妥善管理。因此代理治理本质上是实现对生命周期的控制与可见性——管理方式即如何构建和维护代理，风险管理即如何防范代理的常见故障模式，安全管理即如何控制代理访问权限与可观测性，能够查看代理执行的所有操作。

**[00:02:05]**

数据治理在构建代理系统中扮演什么角色？基本上在构建代理时，你首先进行数据准备，这可能包括创建数据表、创建视图和数据掩码、构建向量数据库，然后开始构建代理。所以你正在将工具组合起来，你正在添加提示词，你正在选择你的 LLM，然后你正在评估代理——通过 LLM 裁判、通过同行标注，或者拥有那个黄金数据集，你可以用它来评估代理的追踪能力。

**[00:02:38]**

部署代理是下一步，确保你的代理已就绪，你正在追踪溯源。从开始到结束是治理层，因此你需要全生命周期管理、风险管理、安全与可观测性。我们将应用这些治理层，借助统一目录（Unity Catalog）和 MLFlow。

正如我之前提到的，治理涵盖你构建的一切，从数据创建到模型监控。所以我希望为开发者提供一种概念化和分类这些领域的办法，这就是我为你制定的四个支柱。

**[00:03:13]**

这些代理治理支柱包括：生命周期管理、风险管理、安全与可观测性，并包含最佳实践。

对于**生命周期管理**，最佳实践是职责分离，这允许多个团队管理数据和模型变更，通过开发、测试和生产环境及版本控制，确保每个阶段的审核。

**风险管理**的最佳实践是纵深防御，实施多层重叠的防御机制，这是你的 PII 检测、护栏机制、合规控制与监控，以保护从数据摄入到模型性能安全的所有环节。

**[00:03:52]**

这里（安全）的最佳实践是最小权限访问，确保代理和用户仅获得其角色所需的最小权限，通过加密实现身份验证和细粒度访问控制。

最后是**可观测性**，能够审计所有操作，这会记录系统交互的完整日志，包括数据访问、模型操作和预测，实现完全可追溯性和合规报告。

**[00:04:18]**

现在我将带你们了解实现各支柱最佳实践的常见技术和工具。

因此在**生命周期管理**和职责分离方面，你将使用：版本控制、CI/CD 流水线、环境管理（这就是当你拥有 staging 和生产环境及隔离机制时）、部署编排、变更管理，以及具备回滚能力。

**[00:04:46]**

关注**风险管理**的纵深防御，这是数据质量监控、PII 检测，这些是你的边界防护、任何合规能力，以及模型验证安全。

**最小权限访问**，这就是你的 SSO 登录，这些是你的 API 密钥、你的多因素认证服务主体、密钥管理访问控制、数据保护和网络安全。

**[00:05:14]**

然后是**可观测性**，这是对生成式 AI 全流程的审计能力。我们经常谈到 OTEL（开放遥测标准）用于模型日志追踪，但这不止于此。这是日志审计记录、应用日志、指纹、日志访问、日志监控能力、血缘追踪与告警。

**[00:05:35]**

这就是四大核心领域。下次你想构建智能体能力时，并思考"这是否已准备好部署"、"这是否适合生产场景"，我已创建了一份检查清单，帮助你牢记这些最佳实践。

例如：

- 能否通过环境安全推广变更并具备审核与回滚能力？
- 是否有多层防护机制在生产前拦截问题？
- 所有数据源，是否仅授权智能体和用户可访问？
- 能否追踪智能体运行时使用的每个工具，以及访问的数据和返回的结果？

**[00:06:16]**

这些是生命周期管理的关键能力、风险管理、安全与可观测性。在本短课程中，你将看到这些支柱的实际应用。
