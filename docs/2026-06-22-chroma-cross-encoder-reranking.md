# 交叉编码器重排序：用精排解决检索噪音（Advanced Retrieval for AI with Chroma p05）

## 一句话总结

交叉编码器通过联合编码查询与文档输出精准相关性分数，无法预计算所以只能作为第二阶段精排器——与多查询扩展组合后，先用双编码器宽召回，再用交叉编码器（始终对原始查询打分）过滤噪音，实现召回率与精确率双赢。

## 核心观点

1. **双编码器与交叉编码器的根本差异决定了各自的使用场景** — 双编码器将查询和文档独立编码为向量，文档向量可预计算存储，检索时仅需一次查询编码，速度快、可扩展到百万量级语料；但独立编码无法捕捉查询与文档间的深度语义交互。交叉编码器将查询+文档拼接后联合编码，精度更高，但无法预计算文档向量——每次推理都必须重新对每个候选文档计算，因此只能用于候选集很小的精排阶段，绝对不能用于全量检索。

2. **"多取再精排"是组合方案的核心逻辑** — 实验一将双编码器的候选集从 5 扩大到 10（`n_results=10`），目的是捕捉那些被余弦距离排到第 6~10 位但实际更相关的"长尾"文档，然后用交叉编码器对这 10 个候选重新打分排序，取分数最高的几个传给 LLM。这一步修正的是"双编码器排序误差"，而不是召回本身。

3. **组合技的关键约束：交叉编码器必须用原始查询打分** — 实验二将多查询扩展（p04）和交叉编码器重排序结合：多个扩展查询分别检索取 top-10，set 去重合并所有候选，然后交叉编码器对所有候选文档打分。关键细节是 `pairs = [[original_query, doc] for doc in unique_documents]`——使用**原始查询**而不是扩展出的子问题。原因：扩展查询的目的是扩大召回覆盖，但最终排序的标准是"能否回答用户的原始问题"，用子问题打分会把偏离原始意图的文档错误提权。

4. **ms-marco-MiniLM-L-6-v2 是轻量级但专用的重排序模型** — 该模型在 Microsoft MARCO 段落重排序数据集上微调，任务就是判断"这段文字是否回答了这个问题"，输出原始分数（可能为负数，非 0~1），分数越高相关性越强。作为交叉编码器，它的参数规模远小于 GPT，推理成本可接受，适合在小候选集上部署。

5. **完整的三层检索流水线：查询扩展 → 双编码器召回 → 交叉编码器精排** — 这条流水线同时解决了 p03 中诊断出的两类问题：多查询扩展解决"语义漂移"（一个查询覆盖不全），交叉编码器解决"结果分散"（余弦距离排序有噪音）。set 去重在两者之间起到连接作用，避免相同文档被重复计分。

## 完整组合流水线

```
原始查询
    ↓ LLM（augment_multiple_query）
5 个扩展子问题
    ↓ Chroma 双编码器（每个 top-10）
~50 条候选（含重复）
    ↓ Python set 去重
唯一候选集（约 20~40 条）
    ↓ CrossEncoder（pairs 始终用 original_query）
相关性分数
    ↓ np.argsort()[::-1][:5]
Top-5 最相关文档
    ↓ LLM
最终答案
```

## 可执行建议

- **n_results 扩大到 10 不是随意选的**：交叉编码器在 10~20 个候选上的额外延迟通常可以接受，但如果候选集超过 50 个，推理时间会显著增加，需要评估延迟预算。生产中常见做法是双编码器取 20~50，交叉编码器精排后取 top-5 或 top-10。
- **组合查询扩展时，去重的效率有限**：`set` 是精确去重（完全相同的字符串），但不同措辞的相似 chunk 不会被去重。如果知识库 chunk 粒度细，可能需要用语义去重（如对 chunk 嵌入做聚类）来真正减少候选数量。
- **交叉编码器分数是原始 logit，无需归一化**：`cross_encoder.predict()` 输出的分数值不可跨模型比较，但同一批次内可以直接用 `np.argsort` 排序，不需要先 softmax 或 sigmoid。
- **ms-marco 模型适合英文问答，中文场景需替换**：如果检索的是中文文档，建议替换为中文交叉编码器模型（如 `cross-encoder/ms-marco-chinese-base` 或类似中文重排序模型），否则精排质量会大打折扣。
- **调试时用 UMAP 可视化验证重排序效果**：将重排序前后的 top-5 文档同时投影到 UMAP，对比绿圈是否更聚集在查询向量附近，可以直观验证重排序是否真的改善了结果。

## 关键术语

| 术语                           | 说明                                                                                  |
| ------------------------------ | ------------------------------------------------------------------------------------- |
| 双编码器（Bi-encoder）         | 查询和文档分别独立编码，相似度用向量余弦距离衡量，文档可预计算，适合大规模召回        |
| 交叉编码器（Cross-encoder）    | 查询+文档拼接后联合编码，输出相关性分数，精度高但无法预计算，只适合小候选集精排       |
| 重排序（Re-ranking）           | 对召回候选集按更精准的相关性重新打分排序，通常是检索流水线的第二阶段                  |
| ms-marco-MiniLM-L-6-v2         | 在 Microsoft MARCO 重排序数据集上微调的轻量级交叉编码器，输出"文档是否回答问题"的分数 |
| `cross_encoder.predict(pairs)` | 对 `[[query, doc], ...]` 列表批量打分，返回原始 logit 数组                            |
| `np.argsort(scores)[::-1]`     | 按分数降序排列原始索引，用于提取重排序后的文档顺序                                    |
| 精排（Re-ranking Stage）       | 检索流水线中接在召回之后的第二阶段，用高精度模型对小候选集重新评分                    |
| 原始查询（original_query）     | 用户实际输入的问题，交叉编码器精排时必须用它打分，而非扩展子问题                      |
