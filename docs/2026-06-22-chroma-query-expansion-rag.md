# 查询扩展：用 LLM 增强检索意图（Advanced Retrieval for AI with Chroma p04）

## 一句话总结

查询扩展通过 LLM 补充原始查询的信息量，用两种方式解决语义漂移：假设答案扩展（HyDE）让查询向量移向文档区域，多查询扩展从多个维度撒网并去重合并——两者都能显著提升 top-k 检索的相关性。

## 核心观点

1. **语义漂移的根源是"查询太短、信息密度太低"** — 用户的短查询（如"Was there significant turnover in the executive team?"）在嵌入空间中携带的信息量远少于包含答案的文档段落，导致向量距离无法准确反映语义相关性。查询扩展的本质是用 LLM 给查询"补血"，让它带着更多信息去检索。

2. **HyDE（假设答案扩展）通过模仿文档语言风格拉近两者距离** — `augment_query_generated()` 要求 LLM 生成"一个可能出现在年度报告中的示例答案"，然后将"原始查询 + 假设答案"拼接为 `joint_query` 再做检索。System prompt 中"might be found in a document like an annual report"是关键限定——让 LLM 用年报语言而非通用解释语言回答，生成的假设答案词汇与真实文档更接近。UMAP 图中橙色 X（扩展后查询）明显向数据云中心移动。论文来源：[arxiv 2305.03653](https://arxiv.org/abs/2305.03653)。

3. **多查询扩展靠"多维度覆盖"提升召回率** — `augment_multiple_query()` 让 LLM 生成最多 5 个相关子问题，每个覆盖原始问题的不同侧面，批量检索后用 Python `set` 去重。与 HyDE 相比，它更适合本身就有多个子维度的复杂问题（如"影响营收增长的最重要因素是什么"），能从知识库的不同区域各取所需，最终覆盖范围更广。

4. **System prompt 的格式控制直接决定代码可用性** — `augment_multiple_query()` 的 prompt 中"Output one question per line. Do not number the questions."是为了让 `.split("\n")` 能可靠解析结果。去掉编号避免了解析时残留"1."、"2."等前缀，是典型的"为下游代码设计 prompt"的工程实践。

5. **两种技术各有适用场景，主要风险不同** — HyDE 适合查询词与文档词不匹配的场景，主要风险是 LLM 假设答案偏离事实（如捏造财务数字），将噪音引入检索；多查询扩展适合需要广覆盖的场景，主要风险是扩展问题偏题以及 token 消耗和延迟增加（N 次检索）。

6. **UMAP 可视化在这一讲继续作为直觉工具** — 新增橙色 X 标记扩展后的查询向量，与红色 X（原始）对比可以直观验证：扩展后查询向量是否移向数据密集区域、检索结果（绿圈）是否更聚集。这让"扩展是否有效"从定性判断变为可视化确认。

## 两种查询扩展技术对比

| 维度       | 假设答案扩展（HyDE）                                 | 多查询扩展                                           |
| ---------- | ---------------------------------------------------- | ---------------------------------------------------- |
| 核心机制   | 原始查询 + LLM 生成的假设答案 → 拼接后检索           | 原始查询 → LLM 生成 5 个子问题 → 各自检索 → 去重合并 |
| UMAP 表现  | 橙 X 向数据云中心漂移                                | 多个橙 X 散布在不同区域                              |
| LLM 调用   | 1 次                                                 | 1 次                                                 |
| 检索调用   | 1 次                                                 | N+1 次（原始 + N 个子问题）                          |
| 最适合场景 | 查询词与文档词不匹配（语义漂移）                     | 复杂多维度问题，需要广覆盖                           |
| 主要风险   | 假设答案捏造事实 → 引入噪音                          | 子问题偏题 + token 消耗高 + 延迟增加                 |
| 参考论文   | [arxiv 2305.03653](https://arxiv.org/abs/2305.03653) | —                                                    |

## 可执行建议

- **HyDE 的 system prompt 必须限定文档风格**："provide an example answer that might be found in a document like an annual report"——如果没有这个限定，LLM 会写通用解释，反而引入无关词汇，降低检索精度。
- **多查询去重用 `set`，但要注意顺序丢失**：`set` 去重高效，但 Python set 不保序。如果后续需要按相关性排序，建议改用带去重的有序结构（如遍历 list 时维护一个 seen set）。
- **多查询扩展的子问题数量不宜过多**：每增加一个子问题就增加一次向量检索调用，实际生产中建议 3 个左右，在召回率提升和延迟之间取平衡。
- **两种技术可以组合**：先用多查询扩展提升覆盖率，再对每个子问题结果用 HyDE 做精排——但这会显著增加 LLM 调用次数，需评估延迟和成本。
- **用 UMAP 做 A/B 对比验证**：对同一个查询，先跑不加扩展的基础检索，再跑扩展后的检索，对比绿圈位置是否更聚集，是低成本的定性验证方式。
- **HyDE 在开放域问答中比在封闭域（特定文档）中风险更高**：微软年报场景中，LLM 对财务数字有一定了解，假设答案不至于完全偏离；但在高度专业的私有知识库（如内部产品文档）场景，假设答案可能严重偏离，需谨慎。

## 关键术语

| 术语                                     | 说明                                                                |
| ---------------------------------------- | ------------------------------------------------------------------- |
| 查询扩展（Query Expansion）              | 在检索前通过 LLM 增强或拆解原始查询，以提升检索相关性               |
| HyDE（Hypothetical Document Embeddings） | 假设文档嵌入，生成假设答案后拼接查询进行检索，论文 arxiv 2305.03653 |
| `augment_query_generated()`              | 课程定义的 HyDE 实现函数，生成一个"可能出现在年报中"的假设答案      |
| `augment_multiple_query()`               | 课程定义的多查询扩展函数，生成 5 个相关子问题                       |
| joint_query                              | 原始查询 + 假设答案的拼接字符串，用于 HyDE 检索                     |
| 去重（Deduplication）                    | 多查询检索后合并结果，用 `set` 移除重复文档块                       |
| 橙色 X                                   | UMAP 图中代表扩展后查询向量的标记（区别于红色 X 的原始查询）        |
| 语义漂移（Semantic Gap）                 | 查询词与文档词不匹配，嵌入向量无法反映真实语义相关性                |
