## 一句话总结

Loop Engineering 是从"你写提示词驱动 Agent"到"你设计系统让 Agent 自驱动"的范式转变，结合 Hermes 的自我进化技能和非交互式 Claude Code，可以实现确定性与非确定性两类全自主循环。

---

## 核心观点

**1. Loop Engineering 不是新概念，但结合持续运行的 Agent 才真正有价值**

Cron Job 就是最朴素的循环。Loop Engineering 的新意在于：用 AI Agent 替代人工在每个步骤之间做错误检查和纠偏——以前"你就是那个循环"，现在 Agent 自己是循环。

**2. 提示词工程 → 循环工程：关注点从"指令"转向"系统"**

提示词工程只控制了五步中的"决定"步骤，而循环工程把检查状态、决策、执行、收集反馈、判断完成这五步全部覆盖。你的工作从写提示词变成定义最终目标。

**3. 模型能力阈值决定了循环成为可能的时机**

Opus 4.5 是真正的转折点，长任务质量大幅提升，不再需要精心设计的"引导架构"（harness）逐步扶持 Agent。Fable 5 则专为长期复杂任务设计，任务越复杂表现越好——与传统模型截然相反。

**4. 五步循环系统**

① 检查项目状态 → ② 决定下一步 → ③ 执行（调用工具/写文件/跑命令）→ ④ 收集反馈 → ⑤ 判断是否完成。循环不完成不退出，Agent 持续迭代。

**5. 搭建有效循环的六个关键要素**

- **上下文管理**：系统提示易被近期工具输出淹没，必须主动管理
- **反馈质量**：测试结果、UI 截图等——Agent 据此判断下一步
- **验证关卡**：把反馈转化为明确的通过/不通过裁决
- **终止条件**：必须显式设定，否则循环太早退出或无进展空转
- **错误处理**：工具失败时的处理预案，避免系统进入损坏状态
- **状态管理**：用外部文件跨轮追踪进度，对抗上下文窗口限制

**6. 两种循环类型：确定性 vs 非确定性**

- **确定性循环**：有明确完成标准（测试通过、代码编译成功）。适合生产监控+自动修复，Hermes 用非交互式 Claude Code 跑测试，全部通过后自动 PR 提交。
- **非确定性循环**：无法用规则验证（UI 质量、功能主观判断）。用"AI Slop 检测器"技能持续检测+修复，技能本身可根据反馈自我更新。

**7. 对抗性循环（Adversarial Loop）提升非确定性任务质量**

Claude 模型负责构建，GPT 模型负责审查——两者互相检查，形成对抗循环。GPT 在代码审查上的优势让最终 UI 质量明显优于单模型输出。

**8. Hermes 的自我进化技能是关键差异化**

Hermes 会基于工作流自动创建和进化技能。若循环结束后仍发现问题，只需指出，技能会直接更新自身纳入新反馈——验证器随使用变得越来越强。

## 可执行建议

1. **立即可做**：在下一个有明确验收标准（如测试集）的任务上，用 `/goal` 设置终止条件，观察 Agent 是否能不中断跑到完成。
2. **本周可做**：把一个每天重复的任务（如日志分析、代码健康检查）改造成 Cron/Routine 触发的确定性循环，用测试结果或状态检查作为终止条件。
3. **进阶：搭建生产监控循环**：在 Hermes 上配置应用监控自动化，触发非交互式 Claude Code 修复，直到所有测试通过后自动 PR。
4. **非确定性任务**：创建一个"质量检查技能"文件（列出你希望避免的模式），让 Agent 在每次输出后对照检查，不达标就继续修；发现新模式时更新技能文件。
5. **引入对抗性循环**：在代码审查或 UI 评估中，用不同模型（或不同提示角色）分别扮演"构建者"和"审查者"，两者互相验证。

## 关键术语

| 术语                   | 说明                                                                 |
| ---------------------- | -------------------------------------------------------------------- |
| Loop Engineering       | 循环工程；让 Agent 自驱动，替代人工在步骤间做错误检查和纠偏          |
| Deterministic Loop     | 确定性循环；有清晰的通过/不通过标准（如测试用例）                    |
| Non-Deterministic Loop | 非确定性循环；无法用规则验证的任务（如 UI 质量）                     |
| Adversarial Loop       | 对抗性循环；两个不同模型（或角色）互相检查对方工作                   |
| Self-Evolving Skills   | 自我进化技能；Hermes 功能，技能文件基于反馈自动更新                  |
| Verification Gate      | 验证关卡；把反馈转化为明确裁决的检查点                               |
| Termination Condition  | 终止条件；告诉循环何时停止的显式规则                                 |
| Non-Interactive Mode   | 非交互模式；Claude Code 独立运行，不需要人来驱动                     |
| Ralph Loop             | 早期循环工程实现；用 Hooks 阻止 Agent 在未满足条件时标记完成         |
| /goal                  | Claude Code 命令；设置任务完成条件，由第二个模型审核是否真正完成     |
| Goal Buddy 2           | 在 /goal 基础上构建；让 Agent 在本地文件追踪进度，开始前就定义"完成" |
| Hermes                 | 持续运行的 Agent；内置自我进化技能，适合实现两类循环                 |
| AI Slop                | AI 生成内容的固定模式痕迹，降低 UI 和内容质量的标志                  |
| Context Rot            | 上下文腐烂；长对话中重要指令被近期输出淹没导致质量下降               |
