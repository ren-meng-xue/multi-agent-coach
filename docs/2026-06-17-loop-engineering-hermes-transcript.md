最近有个新术语流传开来，你可能已经听说过了——叫做"循环工程（Loop Engineering）"。和每一个炒作术语一样，大家都在谈论它，好像它是什么新事物。其实不是。但当你把它和一个持续运行的 Agent（比如 Hermes）结合在一起时，它就不再只是噱头了。

大多数正在尝试搭建这类系统的人，把循环本身搭对了，却遗漏了真正让它跑起来的那个东西。如果你已经知道循环有两种类型，那其中有一种特定的配置方式几乎没人在做。一旦你看到它，你对用 Agent 构建东西的思考方式就会彻底改变。看完这个视频，你会清楚地理解它是什么，并且能把它跑在 Hermes 上——甚至在 Claude Code 上——完全不需要你介入。

**[00:00:38]**

循环工程的核心思路很简单：你不再是那个写提示词来驱动 Agent 的人，而是让 Agent 自己驱动自己。但要理解为什么这是一种转变，你得先把它和之前的东西做个对比。

过去重要的技能是提示词工程（Prompt Engineering），我们所有的精力都花在写出正确的一系列指令来正确地驱动编程 Agent。但循环工程不是让你自己写提示词，而是你来设计那个替你做提示词工程、并自主驱动 Agent 的系统。所以，关注点从"如何精心撰写指令"，转移到了"如何设计能自我运行的系统"。

**[00:01:11]**

这一切的起点，是 OpenClaw 的创建者说的那句话：你不应该再自己给编程 Agent 写提示词了，你应该把精力放在设计那些替你提示 Agent 的循环上。而且他不是唯一这么说的人。Boris（Claude Code 的创建者）在 Anthropic 年度开发者大会上也说了同样的话——他说他不再自己运行提示词了，让 Claude 自己判断需要做什么。

所以问题来了：怎么开始？一切归结于你能多好地搭建那些"你完全不用担心给 Agent 写提示词"的系统。你定义你需要什么，Agent 做剩下的。这正是 AI 驱动开发正在走向的方向。

**[00:01:51]**

在讲如何搭建之前，先把"什么是循环"说清楚。循环，本质上是一个过程：你定义最终目标，Agent 自己找出达成它的步骤，在过程中自我纠正，绕过障碍，直到达成你设定的目标。

几个月前，在模型还没有能力维持长时间任务之前，这根本不可行。当时的做法是：看 Agent 在做什么，自己检查输出，找到问题，再写提示词修复。**你就是那个循环**。你是在每个步骤之间做错误检查和纠偏的那个人。这仍然是大多数人的开发方式。而这正是循环工程即将从你手里拿走的东西。

**[00:02:29]**

这听起来可能像个新概念，但其实循环已经存在很久了。Cron Job 就是你可能已经见过的一种循环——它们只是定时自动重复运行的任务，不需要你每次手动触发。唯一真正的区别是，Cron Job 在固定时间运行。

有了循环之后，工作的重点就不再是写提示词了。你的 Agent 在任务上的表现，取决于你对最终目标的定义有多清晰。

对你们中的一些人来说，这个过程听起来很像强化学习（Reinforcement Learning）。如果你没接触过，简单说：强化学习是一种训练模型的方式，你不告诉它正确答案，而是在它做对时给正向信号，做错时给负向信号，让它自己找出正确路径。同样的思路在这里也适用，只不过被训练的不是模型本身，而是 Agent 在完成你想要的任务时不断迭代——就像模型在训练中不断改进一样。如果 Agent 没通过你设置的循环检查，它不会把任务标记为完成，而是再试一次，持续下去，直到达成你设定的目标。

**[00:03:33]**

听完这些，你可能会想：如果一切都在走向自主，那还剩什么是你来做的？但你的角色不会缩小——反而会变得更重要，因为正是你的领域知识和经验决定了最终目标本身。而这会体现在你构建和发布的每一个东西上。

这也正是为什么向自主循环的推进只会加速，并且体现在现在每一个新功能的发布上。**Fable 5** 是迄今为止最清晰的例子。Anthropic 在呼吁放缓 AI 开发的同时，还是发布了它——因为模型进步的速度很难跟上。在发布一段时间后他们甚至撤回了它。它是为长期、复杂任务而生的，而且任务越长越复杂，它的表现越好——这基本上和以前模型的工作方式相反。

**[00:04:17]**

这次转变真正始于 Opus 4.5。那个版本发布后，长时间运行的任务质量大幅提升，你不再需要用精心设计的"引导架构"（harness）——基本上是手把手引导 Agent 完成每个步骤的结构化配置——来帮 Agent 跑任务了。关注点转向了如何让项目做好长期运行的准备，因为模型现在已经有能力自己处理事情，不需要太多逐步引导。

但循环本身不是唯一重要的事。你还需要以一种让 Agent 能长时间独立工作的方式来组织你的项目。因此，很多人开始搭建并开源这类系统。

**[00:04:51]**

**Ralph Loop** 是最早的之一。它通过设定最终目标并确保 Agent 不会偏离来工作。它用的是 Hooks——在特定事件触发时自动运行的脚本——来阻止 Agent 把任务标记为完成，除非它真的满足了条件。但 Hooks 比较僵硬。所以 Claude 引入了自己的 `/goal` 命令，做了同样的事，但更灵活——它不是硬编码的检查，而是让另一个模型来判断任务是否真的完成了。

我们之前介绍过的 **Goal Buddy 2** 在此基础上更进一步：让 Agent 在本地文件里追踪进度，并在开始之前就精确定义"完成"是什么样子，这样它始终知道自己在朝什么目标努力。**Hermes Agent** 和 **Open Claw** 都建立在同样的理念上：把你从整个过程中移除，让 Agent 自己处理一切。

**[00:05:38]**

现在，如果你想开始搭建，这里有一个简单的五步系统。由于循环有两种类型，某些步骤的工作方式略有不同，稍后我们会讲到两种类型。现在先从 Claude Code 开始，之后再看在 Hermes Agent 里怎么做同样的事。

**五步循环系统：**

1. **检查项目状态**——模型从当前状态判断下一步应该是什么
2. **决定下一步行动**
3. **执行**——实际工作发生的地方：Agent 调用工具、写文件、运行命令来完成任务
4. **收集反馈**——看看实际发生了什么
5. **判断是否完成**——基于反馈决定任务是否达成目标

在这五步里，提示词工程和循环工程的差别就显而易见了：提示词工程你只控制了"决定"这一步，而循环工程把全部五步都处理了。

**[00:06:22]**

**搭建一个有效循环需要做对以下几件事：**

**上下文管理（Context Management）**：关注每一轮进入上下文的内容，因为这决定了 Agent 在任意时刻实际知道什么。你不能指望对话历史无限扩展，哪怕上下文窗口大到一百万 token——因为随着对话增长，你的系统提示和指令会被最新的工具输出掩盖。Agent 的注意力自然会被最近的内容吸引，重要的东西就丢失了。这就是上下文管理如此重要的原因。

**反馈质量（Feedback Quality）**：反馈告诉 Agent 它做得怎样，是整个系统中最重要的信号之一。它可以有很多形式，比如测试运行的输出，或者它刚构建的 UI 的截图。无论是什么形式，那就是 Agent 读取来判断下一步的东西。

**验证关卡（Verification Gates）**：把反馈转化为明确的裁决。它们是告诉 Agent 任务是否真的完成了的检查点。

**终止条件（Termination Condition）**：一条告诉循环何时停止的规则。这个必须明确设定，否则 Agent 要么太早退出，要么在没有实质进展的情况下一直跑下去。

**错误处理（Error Handling）**：最常被忽视的一点。你必须告诉模型当工具调用失败时该怎么做，这样系统能干净地处理，而不是留下一堆残局制造更多问题。

**状态管理（State Management）**：随着对话增长，追踪任务进度。上下文窗口不能永远保存所有内容，所以要依赖外部文件来为 Agent 追踪信息，让它在不丢失进度的情况下持续工作。

**[00:08:01]**

不过，由于你是把"找出路径"的工作交给了模型而不是自己来做，循环在 token 上会很贵。所以你需要在什么时候真正用它这件事上保持审慎。循环能用的 token 越多，它处理任务的效果往往越好。

## 两种循环类型

**[00:09:20]**

循环有两种类型。

**第一种：确定性循环（Deterministic Loop）**

这适用于有明确"完成"定义的任务——比如测试通过、代码成功编译，或者类似的东西。这类循环相对直接，因为最终目标清晰，模型确切知道在宣告任务完成之前需要做什么。

由于 Hermes 是持续运行的，它非常适合实现这类循环。之前的视频里我们展示过如何在它上面创建多个工作流。

**[00:09:52]**

确定性循环的核心是对最终目标的清晰定义。对于你部署的应用来说，那个定义就是你的测试用例。所以你可以把 Hermes Agent 指向任何你部署了测试用例的应用，让它帮你监控。如果某次变更或提交导致生产环境出问题，你可以在 Hermes 上设置一个自动化来捕捉它。

这里效果特别好的原因是 Hermes 带有**自我进化技能（Self-Evolving Skills）**功能——它会基于工作流自动创建和进化技能，持续维护应用的健康状态。

一旦你设置好了那个监控自动化，你可以让它以**非交互模式**启动 Claude Code——基本上就是不需要你来驱动的独立运行模式——然后让它在循环里修复问题，直到所有测试用例通过。它在这之后做的是：设置自动化工作流，加载"子 Agent 驱动开发"技能和"GitHub PR 工作流"技能——这些技能告诉它如何在 GitHub 上管理应用。它先找出导致生产故障的问题，然后以非交互模式启动 Claude Code，跑测试，等所有测试通过后提交改动。在跑完所有测试并修复了导致生产失败的原因之后，它用 GitHub CLI 提交改动。最终应用运行无故障，因为它已经确认了成功部署所需的所有检查都已通过。

**[00:11:22]**

**第二种：非确定性循环（Non-Deterministic Loop）**

这适用于那些没有明确规则来检验任务是否完成的情况——不像确定性循环那样能设定清晰的检验规则，因为没有干净的方式来验证结果。这类任务是我们作为人类可以自己观察和判断的——比如构建 UI，或者实现一个需要主观判断的功能。

**[00:11:44]**

在非确定性循环中，工作流有所不同。如果你在 UI 上用 AI，你已经知道它倾向于反复套用同样的模式。这就是为什么我们创建了一个叫做 **"AI Slop 检测器"** 的技能，里面包含了如何避免"AI 味道"的所有指令，并列出了最能暴露 AI 生成痕迹的模式。

我们再次使用 Hermes 的原因也是它的自我进化技能：如果我们在运行这个技能后的 UI 里仍然发现 AI 味道，技能可以直接更新自身来纳入那些反馈。这正是我们为这个工作流做这个设置的原因——让技能检查 UI 是否有那些模式，如果有就修复，然后以非交互模式启动 Claude Code 运行技能，持续修复发现的问题，直到没有问题为止。

**[00:12:27]**

Hermes 还带来另一个好处：审查工作的模型和构建工作的模型是不同的。我们使用 GPT 模型（在代码审查方面公认是最强的之一）。所以 Claude 模型负责构建，GPT 模型负责审查——这构成了一个**对抗性循环（Adversarial Loop）**，两者互相检查对方的工作。

那个循环跑完之后，生成的 UI 比现在 Opus 模型普通输出的质量好得多。而且如果在 Agent 循环结束后你仍然发现任何 AI 味道，只需提一下，它会替你更新技能，加强你已有的验证器。我们已经集体强化了这个技能，涵盖了我们和 Hermes 共同识别出的多种 AI Slop 模式。
