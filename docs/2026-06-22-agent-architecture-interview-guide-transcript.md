**[00:00:00]**

大家好，这里是 AI 深度幻觉。今天的面试题是：一个完整的 agent 智能体架构一般包括哪些部分？这题坑特别多，我见过很多人上来就背名词——感知、推理、记忆、工具、编排——背完面试官追一句，你就愣住了。面试官想听的不是名词列表，而是你对这些组件的理解深度。我们今天从教科书到生产实战一层一层拆。

先把教科书答案摆出来再说哪些是面试官真正想听的。教科书会告诉你七大核心组件：感知层负责接收输入，推理引擎是大脑，记忆系统分短期、长期和情景三层，工具使用连接外部世界，编排管理多步骤工作流，multi-agent 是可选的多智能体协作，最后是安全和治理。

**[00:00:48]**

这七个答出来能及格吗？能及格但不出彩。因为面试官紧接着会问：那你觉得推理引擎应该用 ReAct 还是 Plan-and-Execute？这时候如果你只会背定义就露馅了。

等一下，这几个模式我背得滚瓜烂熟，你是说背出来也没用？不是没用，而是你得知道它们在生产里到底长什么样。

**[00:01:09]**

从生产实际来看，这些都不是独立的复杂架构，它们都是同一个 while loop 的不同变体。2026 年 3 月，Claude Code 51 万 2000 行源码泄露，给了我们极其罕见的一手证据。Claude Code、Cursor、Manus、Devin 四个产品核心架构居然是一样的。

那花几十亿做的东西跟几行伪代码有什么区别？区别大了，但不在循环本身。我们之前在《从 Claude Code 源码看 Agent 架构之争》那期详细拆过——核心就是模型调用加工具列表，有 tool call 就执行，没有就结束循环。但 51 万 2000 行代码里，99% 都是围绕这个循环做的工程权限系统、错误恢复等。

**[00:01:53]**

那 ReAct、Plan-and-Execute 这些学术名词在生产中到底对应什么？

**ReAct** 就是给这个 while loop 加了一个显式的思考步骤。有博主在描述 Cursor 架构时直接说，ReAct 只是实现这个循环的一种常见方式。

**Plan-and-Execute** 在生产中也不是严格的先规划后执行。Claude Code 和 Manus 都是用 TODO 文件做"软规划"，规划和执行交错进行。

所以 Anthropic 自己也承认 agent 其实很简单，原话是"agent 的实现往往很简单，就是 LLM 在循环中基于环境反馈调用工具"。他们甚至说最成功的实现不用复杂框架，而是用简单的可组合的模式。还有工程师补了一句：复杂度会杀死迭代速度。

**[00:02:39]**

那面试的时候直接说"agent 就是个 while loop"，不会显得太简单了吗？面试官会不会觉得你在敷衍？

不会，因为接下来你要告诉面试官：loop 虽然简单，但真正决定 agent 好坏的是工程细节——工具怎么设计、上下文怎么管理、怎么编排，这才是水面下的冰山。

**[00:02:59]**

**工具设计为什么排第一？比模型选择还重要吗？**

有一手数据：Anthropic 在 SWE-bench 上，用升级版 Claude 3.5 Sonnet 配合仅两个工具——bash 和 edit——再加上精心优化的工具描述，就刷到了 49%。当时 Anthropic 的原话是："给模型设计工具接口，应该投入跟给人设计界面一样多的精力。"

只靠两个工具就刷到那个分数，那是不是说工具不在多而在精？完全正确。而且工具太多反而有害——SpeakEasy 团队做过测试：小模型在 10 个工具时完美，20 个时差一道题，但到了 107 个，大小模型都开始频繁幻觉和出错，不是线性退化，是悬崖式崩塌。

GitHub Copilot 好像也验证了这个？对，Copilot 把核心工具从 40 个精简到 13 个，配合智能路由，基准提升了 2~5 个百分点。现在常规模型 SWE-bench 已经到 80% 左右了，但"工具描述决定性能天花板"这个洞察到今天依然成立。

**[00:04:02]**

那 ACI 具体怎么指导工具设计？

ACI 就是 Agent Computer Interface，SWE-agent 论文原创，Anthropic 借用过来类比 HCI 应用到工具设计上。在《Writing tools for agents》博客里列了五条工具设计原则：选对工具、要为 agent 重新设计而不是包装现有 API、命名空间化、返回语义化信息而不是技术标识符、优化 token 效率，最后是精心编写工具描述——Anthropic 自己说这是最有效的方法之一。

我们之前在《面试官问 MCP 还是 CLI》那期也从协议层面聊过相关问题。但现实中用户会往 Agent 里塞几百个 MCP 工具怎么办？总不能不让人装吧？

有一个产品的解法很优雅——它不删除工具定义，因为那会让 KV cache 失效，而是在解码时用 logit masking 遮蔽不可用工具的 token。工具命名故意用一致前缀，所有浏览器工具都叫 `browser_` 开头，通过前缀预填就能限制在特定工具组内。

**[00:05:06]**

好，工具聊透了，下一个是上下文管理——这是 agent 的隐藏战场。

感觉大家都在聊模型能力，所以这是面试加分项。Anthropic 提出了"context engineering"这个概念，核心是找到最小的高信号 token 集合，最大化期望结果的概率。因为所有模型都有 context rot——token 越多，模型回忆信息的能力越差。

ICML 2025 的 NOLA 测试发现，13 个 LLM 中有 11 个在仅 32000 token 时，性能就降到自身短上下文基线的一半以下。

**[00:05:42]**

我们在《为什么 Claude 总失忆》那期对比过 agent 记忆的两大流派——compaction 和 pruning。今天补充一个关键新数据：JetBrains 在 NeurIPS 2025 上做了实验，**Observation Masking**——就是用占位符替换旧的工具输出，但保留推理历史——跟完全不管理上下文相比，便宜了 52%，解决率还高 2.6%。

等等，那 LLM 做摘要呢？直觉上摘要应该更好啊，毕竟它理解内容。反直觉的是：LLM 摘要反而让 agent 多跑 13%~15% 的步骤，因为摘要平滑了一些本该触发停止的信号，agent 就在那绕圈。

最佳方案是混合策略：Masking 做一线防御，偶尔触发 LLM 摘要兜底。

**[00:06:23]**

那 Claude Code 现在有超大上下文窗口了，还需要压缩吗？窗口大了不代表不需要压缩，context rot 是结构性问题。Claude Code 依然用分层压缩：先清旧工具输出，再总结对话，保留架构决策和未解决的 bug。子 agent 的隔离也是关键——每个子 agent 在独立上下文里干活，只返回 1000~2000 token 的精炼摘要，主对话完全不膨胀。

**[00:06:50]**

好，架构组件和工程细节都聊了，面试还经常问：agent 到底有几种类型？这题怎么答不踩坑？

这是最容易踩坑的问题，分类列表五花八门，是因为分类维度被混在了一起。Databricks 直接说：有的按决策行为分，有的按系统架构分，有的按应用角色分，混在一起讨论类型数量就无限膨胀。

所以面试的时候应该先说清你在用哪个维度分类——这就是关键。

**[00:07:15]**

最实用的维度是 Anthropic 的编排架构三层：

第一层是"增强型 LLM"：LLM 加上检索、工具和记忆。

第二层是 workflow：通过预定义代码编排多个 LLM 调用，有五种模式。

第三层才是真正的 agent：LLM 自主决定流程和工具使用，就是我们说的 while loop。

那 Russell 和 Norvig 的经典五类呢——reflexive、model-based、goal-based 那些，面试要不要提？开场提一句表示你知道经典理论就行，然后迅速转到编排架构维度，因为现代 LLM agent 天然就是 goal-based 加 learning 的混合体，经典分类区分度不高。

**[00:07:54]**

那如果面试官追问：不同类型的 agent 在工程实现上到底差在哪？

骨架完全一样，差异体现在三个配置旋钮：

**第一个是工具集**，这是最大的差异来源——coding agent 用文件读写和终端，客服 agent 用数据库查询和退款接口，Research agent 用搜索和爬虫。

**第二个是 system prompt**，它不只是指令，还是权限系统。VS Code 多 agent 系统就是同一个骨架，plan 模式只读不能改代码，implementation 模式有完整编辑权限。

**第三个是编排模式**——我们在字节跳动 DeerFlow 2.0 那期聊过，单 agent 在大多数场景比多 agent 更可靠。

**[00:08:33]**

所以总结一下面试回答的核心思路——五层递进：

**第一层**：给整体框架，七大组件——感知、推理、记忆、工具、编排、multi-agent 协作、治理。

**第二层**：揭示现实，生产阶段 agent 的核心就是 while loop 加 tool call，ReAct 之类的是学术命名不是独立架构。

**第三层**：深入工具设计，引用 ACI 和 SWE-bench 的数据，这是性能天花板。

**第四层**：context rot 压缩策略，子 agent 隔离。

**第五层**：点明 agent 类型的本质——骨架相同，差异在工具集、system prompt 和编排模式三个配置旋钮。

打到哪一层算及格？达到第一层只能说你背了书，达到第三层面试官会觉得你真的理解 agent，达到第五层面试官可能反过来想跟你讨论了。

**[00:09:26]**

最后说说避坑。我觉得最大的坑可能是过度吹嘘多 agent？你说到点子上了，三个坑：

**第一**，不要只列名词不解释机制。

**第二**，不要把 agent 和 chatbot 混为一谈，agent 的核心区别是自主循环和工具使用。

**第三**，就是你说的——不要过度吹嘘多 agent，Anthropic 和 OpenAI 都明确说过：先穷尽单 agent 能力，再考虑多 agent。
