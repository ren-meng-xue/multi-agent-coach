关于 RAG 系统，你如何处理检索和生成？如何确保检索信息准确且高效辅助生成？ 入库流程：
创建知识库 → 异步抓取/解析 → 生成全局摘要与标题 → Markdown 结构化切分 → Embedding → 写入 pgvector。

问答流程：
Top3 Scope 路由 → Query Rewrite → 向量召回 + FTS 召回 → 合并去重 → Cohere 重排 → Prompt 构建 → GPT-4o 生成 → Citation Guard 校验 → SSE 返回。

优化方式：
准确性通过混合检索、重排阈值、引用校验保证；效率通过 Scope 限定、Top-K、候选集重排和 Query Rewrite 提升。
使用 pgvector 时如何处理向量化输入？是否做预处理或选模型？Cohere 重排标准是什么？ 向量化前先做结构化切分，保持标题、列表、表格、代码块的原子性。生成向量时拼接“章节路径 + 内容”，增强上下文感知。

模型使用 text-embedding-3-small（1536维），批量生成 BATCH_SIZE=100。
向量检索按 cosine_distance 排序，按 Top-K 返回。
重排使用 Cohere rerank-v3.5，Top-N=5，并通过 RAG_MIN_RERANK_SCORE 过滤低相关候选；重排失败则降级为原顺序。
使用框架时遇到准确性或性能问题如何解决？ 主要通过链路策略优化：
1. 引入混合检索，解决术语和错误码召回不稳。
2. 使用 Query Rewrite，解决多轮上下文指代。
3. 引入延迟路由，避免首问闲聊误路由。
4. 宏观问题优先走知识库摘要。
5. 引入 Citation Guard，防止无引用答案。
6. 通过 RAG telemetry 记录召回量、耗时和错误码，定位搜不到、搜错、缺引用、延迟高等问题。
